# Compact Hierarchical Memory Engine (CHME): A Technical-Academic System Note

## Abstract
This document describes the technical design of CHME as a compact, deterministic, in-memory memory tool for document-grounded question answering pipelines. CHME is designed as a tool-first subsystem (not a monolithic RAG stack), exposing stable APIs for ingestion, indexing, retrieval, prompt construction, LLM invocation, and snapshot persistence. The architecture prioritizes low operational complexity, deterministic behavior, and integration ergonomics for downstream assistants, bots, and application backends.

## 1. Problem Definition
Many practical assistants require a local, explainable memory substrate that can:
- ingest heterogeneous Markdown documentation,
- preserve document hierarchy,
- retrieve relevant evidence deterministically,
- support multi-domain partitioning (collections),
- and remain lightweight without external databases.

CHME addresses this by building an explicit hierarchical memory graph and lexical inverted index in memory, then orchestrating retrieval and answer generation through a single `MemoryEngine` interface.

## 2. System Model
CHME operates over a corpus \( \mathcal{D} \) of Markdown files and constructs:
- a set of collections \( \mathcal{C} = \{C_1, \dots, C_m\} \),
- per-collection rooted trees (document root \(\rightarrow\) section \(\rightarrow\) chunk),
- a global engine-level forest across collections.

Each node is represented as:
\[
\text{Node} = (id, text, parent, children, depth, docId, tokens, embedding?)
\]
where `embedding` is optional and unused in current retrieval.

Depth semantics:
- `depth = 0`: document root
- `depth = 1`: section
- `depth = 2`: chunk (retrieval unit)

## 3. Core Data Structures
At collection scope:
- `documents: Map<string, {id, path}>`
- `nodes: Map<string, Node>`
- `keywordIndex: Map<string, Set<string>>`

This is an adjacency-list representation of rooted trees:
- `parent` pointer for upward traversal,
- `children[]` for downward traversal.

The index maps token \(\rightarrow\) chunk-node-id set, enabling efficient lexical candidate generation.

## 4. Ingestion Pipeline
Input: filesystem path containing `.md` files (recursive traversal).

### 4.1 Parsing
Markdown headings (`#`, `##`, `###`) are parsed into hierarchical sections. Plain text files fallback to a synthetic single section.

### 4.2 Chunking
Sections are split at:
- max chunk size: 800 chars,
- overlap: 100 chars,
- preferred boundaries: sentence/newline, then whitespace fallback.

This defines a sliding-window segmenter with overlap:
\[
s_{i+1} = \max(e_i - 100,\ s_i + 1)
\]
where \(s_i\) and \(e_i\) are start/end offsets of chunk \(i\).

### 4.3 Tokenization
Chunk tokens are generated by:
1. lowercasing,
2. punctuation removal,
3. whitespace split,
4. stopword filtering,
5. uniqueness via set.

Only chunk nodes are inserted into `keywordIndex`.

### 4.4 Deterministic IDs
- root: `{docId}:root`
- section: `{docId}:section:{sectionIndex}`
- chunk: `{docId}:{sectionIndex}:{chunkIndex}`

Deterministic IDs guarantee stable references across runs for unchanged inputs.

## 5. Retrieval and Ranking
Given question \(q\):
1. tokenize \(q\) with the same lexical normalizer,
2. gather candidate chunk IDs via inverted index,
3. fallback to all chunk nodes if no candidate exists,
4. score each chunk by keyword overlap:
\[
\text{score}(x, q) = |T_x \cap T_q|
\]
5. sort by descending score, tie-break by lexicographic `node.id`,
6. apply section-aware round-robin selection for diversity.

The result is deterministic under fixed corpus and configuration.

## 6. Context Construction
Top-ranked chunks are rendered into prompt context blocks with section headings:
- `## <section title>` or fallback `## Section`.

Global ask mode additionally annotates source collection:
- `### Collection: <collectionId>`.

A context budget \(L\) (characters) is enforced with block-boundary preservation and sentence-aware truncation if the first block exceeds \(L\).

## 7. LLM Integration Layer
`callLLM` supports:
- local provider (default),
- OpenAI-compatible chat endpoint.

Behavioral properties:
- deterministic intent when `temperature = 0`,
- timeout control (default 30s),
- graceful failure contract (empty string fallback on call/parse failure).

The model call is strictly downstream of retrieval and prompt construction; core memory logic does not depend on model availability.

## 8. Orchestration via MemoryEngine
`MemoryEngine` centralizes:
- collection lifecycle,
- manual and automatic ingestion,
- global/scoped ask dispatch,
- routing rule management,
- retrieval and prompt building APIs,
- runtime LLM configuration (provider/model/urls/keys),
- snapshot save/load.

This creates a single integration surface for external applications and removes the need to import lower-level modules directly.

## 9. Auto-Collection Routing
For large corpora, `ingestAuto` performs deterministic assignment:
1. explicit regex routing rules (priority aware),
2. first-folder path slug fallback,
3. default collection fallback (`general`).

Question-time routing computes per-collection lexical match scores and selects top-N collections with deterministic tie-break.

## 10. Snapshot Persistence
CHME supports compressed JSON snapshots (`.chme.json.gz`) with:
- per-collection state files,
- engine manifest (`_engine.chme.json.gz`),
- schema versioning,
- source metadata (path, size, mtime),
- optional freshness-based partial re-ingest.

Load strategy is replace-existing, enabling fast startup for large corpora while preserving deterministic state restoration.

## 11. Computational Characteristics
Let:
- \(N\): number of chunk nodes in a collection,
- \(V\): unique vocabulary size,
- \(k\): query token count.

Approximate costs:
- Index insert during ingest: \(O(\text{total chunk tokens})\)
- Candidate generation: \(O\left(\sum_{t \in T_q} |\text{postings}(t)|\right)\)
- Scoring: \(O(|\text{candidates}| \cdot k)\)
- Ranking: \(O(|\text{candidates}| \log |\text{candidates}|)\)

Space:
- Nodes: \(O(N)\)
- Inverted index postings: \(O(P)\), \(P\)=total token-node associations.

## 12. Problem Coverage and Practical Utility
CHME is effective when the target problem needs:
- auditable retrieval paths,
- deterministic and reproducible memory behavior,
- low-dependency deployment,
- modular insertion into existing agent/RAG/informal bot stacks.

It is especially suitable as a memory middleware component where teams want control over data flow and evidence traceability before adding heavier semantic components.

## 13. Current Boundaries
Not included in current core:
- embedding-based semantic retrieval,
- learned rerankers,
- external vector/database backends.

These are additive future layers; the current lexical-hierarchical core is intentionally compact and stable.

## 14. Conclusion
CHME implements a disciplined memory architecture built on hierarchical node graphs, lexical indexing, deterministic routing/retrieval, and snapshot persistence. Technically, it solves the engineering problem of reliable document memory orchestration under low operational overhead, while remaining extensible for future semantic upgrades.
